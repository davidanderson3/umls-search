h1. Why Definitions Matter in UMLS Search

I found that the official UMLS API search is driven mainly by concept names, synonyms, and code-linked strings. In many cases that works well, especially when a user searches with terminology already present in the Metathesaurus. Adding definitions does not fundamentally change the behavior of search, but it does provide a small, measurable recall improvement for some short descriptive queries that do not line up cleanly with the canonical label.

To measure the effect of indexing definitions, I compared two versions of the same search stack:

* *Definition-aware search*: the local Elasticsearch index with *definitions* included in ranking.
* *Name/synonym-only baseline*: the same query logic with definition clauses removed. This approximates the behavior of a search system that cannot retrieve on MRDEF content.

The examples below show the pattern I wanted to isolate: indexing definitions can recover some otherwise-missed descriptive queries, producing a modest but measurable recall gain.

h2. What Changes

* With definitions indexed, descriptive phrase queries can match the explanatory language used in MRDEF.
* Without definitions, some of those same queries have weaker lexical overlap with preferred names or synonyms.
* The effect is not dramatic, but it is measurable.
* The clearest gains appear for short clinical descriptions, especially phrases describing a condition, abnormality, inflammation, retention, or loss.

h2. Method

I used a simple ablation study to identify queries where indexing UMLS definitions materially improves search.

# Start with the same local UMLS search index and ranking pipeline in two modes:
#* *Definition-aware search*: normal ranking with MRDEF-derived definition text indexed and queried.
#* *Name/synonym-only baseline*: the same search logic, but with the definition clauses removed.
# Mine candidate queries from concept definitions rather than from preferred names.
# Filter those candidates aggressively so they look like plausible user-entered English queries.
# Run each query against both systems.
# Compare the returned result lists with definitions versus without definitions.
# Keep only cases where definitions clearly help.

h3. Candidate Query Generation

Candidate queries were generated from indexed UMLS definition text, not from preferred names or synonyms. My goal was to find short descriptive phrases that a user might type when they do not know the canonical medical term.

The mining step applied several filters:

* English only
* Maximum of 4 words
* Prefer short clinical-description phrases
* Reject HTML-heavy definitions
* Reject non-clinical semantic types
* Reject awkward fragments, metadata-like text, and obviously incomplete phrases

h3. Ranking Comparison

For each candidate query, we searched the same index twice:

* *With definitions*: definition fields included in ranking
* *Without definitions*: definition fields excluded from ranking

I then compared the returned result lists to see whether the query produced a materially better concept match when definitions were enabled.

h3. Inclusion Criteria

Examples were kept when they met all of the following:

* The query returned a clearly relevant concept when definitions were enabled
* The same query returned no result or a materially worse result without definitions
* The query was short, readable, and plausible as a user-entered search
* The query was not just a duplicate or awkward fragment

h3. Interpretation

This method isolates the contribution of definition text. Because the ranking logic is otherwise unchanged, any improvement in retrieval can be attributed to the presence of definitions in the searchable index.

h2. Takeaway

Definitions add a modest amount of useful recall. They help with a narrow class of short descriptive searches that are semantically correct but do not use the exact preferred UMLS term. In these cases, indexing MRDEF content can recover concepts that are otherwise missed in a name/synonym-only baseline.
